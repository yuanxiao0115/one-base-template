# Scope Guard

## 第一原则

**规则来源不等于规则作用域。**

某条规则即使最初从 `UserManagement` 中总结出来，只要后续会横向推广到其他 `*Management` 模块，就不应继续当作 `UserManagement` 私有经验维护。

## 首版 in-scope

以下内容属于本 skill 首版覆盖范围：

- `apps/admin/src/modules/*Management/**` 下的列表页、搜索表单、CRUD 弹窗、批量操作
- 页面编排瘦身
- 远程搜索、弹窗初始化、首屏加载链路的性能与稳定性修正
- 可横向复用的状态分组、错误反馈、请求收口规则
- 横向推广后需要补的 `docs/plans/` 与 `apps/docs` 说明

## 首版 out-of-scope

以下内容不应直接套用本 skill：

- `apps/portal/**`
- `packages/core/**`
- `packages/ui/**`
- `packages/adapters/**`
- 启动链路、主题系统、路由内核、适配器协议这类跨包基建改造
- 明显绑定某个业务域的数据结构与交互细节

## 可上提的规则

以下规则通常应上提到 `apps/admin/AGENTS.md` 或 docs：

- 页面只保留编排，字段映射与副作用下沉
- 列表 / 弹窗 / 远程搜索的竞态防护方式
- 首屏只阻塞关键数据，非关键数据异步补齐
- 状态分组语义（如 `table / editor / options / actions`）
- 不要用来源模块私有业务名词命名通用能力
- 验证门禁与文档同步要求

## 不应上提的规则

以下规则通常应继续留在模块私有语境：

- 特定业务域接口字段和接口编排
- 某个弹层的独特布局结构
- 某个组织树、角色树、人员选择器的业务特化规则
- 只在单个模块内成立的展示文案与交互细节

## 发现越界时怎么处理

### 情况 1：目标目录不在 `*Management`

停止使用本 skill，改按实际目录范围重新定界。

### 情况 2：需求已经变成跨包重构

把本 skill 只当作“管理模块页内标准化”参考，不继续拿它主导整体方案。

### 情况 3：来源模块经验与其他模块不兼容

保留“规则不成立”的结论，不要硬推广；必要时把限制条件补到更高作用域说明中。
