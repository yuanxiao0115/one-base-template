# 业务接入版本矩阵与迁移模板

> TL;DR：用一张矩阵管理“哪个业务项目使用哪个 major 版本”，再配合迁移模板控制升级节奏。
> 适用范围：接入 `@one-base-template/*` 包的业务项目（外部仓库或子仓库）。
> 最后更新：2026-04-02。

## 1. 背景与目标

- 背景：当多个业务项目接入同一套基础包时，升级窗口与风险承受能力不同。
- 目标：通过版本矩阵与迁移模板，明确责任人、支持窗口和升级计划，避免“谁都以为别人会升级”。

## 2. 范围与非范围

### 2.1 In Scope

- 业务项目版本台账。
- major 版本支持状态（ACTIVE/LTS/EOL）。
- 升级计划与迁移验收清单。

### 2.2 Out of Scope

- 业务项目内部详细改造设计。
- 发版流水线实现细节。

## 3. 前置条件

- 已有版本治理基线：参考 [子包版本治理 SOP（多主线）](/guide/package-version-governance)。
- 每个业务项目已指定维护 owner。
- 业务项目依赖使用 registry 版本号（非 `workspace:*`）。

## 4. 版本矩阵模板

建议按季度维护，至少每月更新一次：

| 项目   | 业务域  | 当前版本线 | 依赖锁定示例                    | 维护状态 | 负责人 | 下次升级窗口 | 备注               |
| ------ | ------- | ---------- | ------------------------------- | -------- | ------ | ------------ | ------------------ |
| A 项目 | OA 后台 | 1.x        | `@one-base-template/ui: ^1.4.0` | LTS      | 张三   | 2026-06      | 历史项目，先保稳定 |
| B 项目 | 新门户  | 2.x        | `@one-base-template/ui: ^2.1.0` | ACTIVE   | 李四   | 2026-05      | 主线跟进           |

维护规则：

1. 一个项目同一阶段只跟一个 major，禁止跨 major 混装。
2. 若项目仍在 `1.x`，必须在矩阵里写明“迁移阻塞项”和目标时间。
3. 进入 EOL 的版本线必须标红并公告冻结日期。

## 5. 迁移模板（1.x -> 2.x）

### 5.1 迁移计划模板

```md
# <项目名> 1.x -> 2.x 迁移计划

- 负责人：
- 目标版本：
- 计划窗口：
- 回滚负责人：

## 范围

- In Scope：
- Out of Scope：

## 风险清单

-

## 实施步骤

1. 升级依赖并安装
2. 处理 breaking changes
3. 回归关键链路
4. 灰度发布

## 验收标准

- typecheck/lint/build 通过
- 核心业务链路回归通过
- 监控指标无异常
```

### 5.2 依赖升级示例

```bash
pnpm add @one-base-template/ui@^2.0.0 @one-base-template/core@^2.0.0
pnpm install
```

### 5.3 验收命令建议

```bash
pnpm typecheck
pnpm lint
pnpm build
```

## 6. 日常维护节奏（建议）

1. 每周：检查 ACTIVE/LTS 项目的已知问题与升级阻塞。
2. 每月：更新版本矩阵并评估 EOL 计划。
3. 每季度：复盘迁移成本，调整下一阶段主线策略。

## 7. 回滚方案（如适用）

- 触发条件：升级后核心链路失败或线上异常升高。
- 回滚步骤：
  1. 将业务项目依赖回退到上一个稳定版本。
  2. 记录失败原因，形成修复项。
  3. 重新评估升级窗口。
- 回滚后验证：核心链路恢复、告警回落、关键页面可用。

## 8. FAQ

| 问题                                           | 原因                                  | 解决方案                |
| ---------------------------------------------- | ------------------------------------- | ----------------------- |
| 为什么要维护矩阵，直接看 package.json 不行吗？ | `package.json` 看不到维护状态与责任人 | 用矩阵补齐“治理信息”    |
| 项目很多，矩阵会不会难维护？                   | 不设模板会更混乱                      | 固定字段，月度例行更新  |
| 什么时候允许继续留在旧版本？                   | 仍有业务阻塞但可控                    | 标注 LTS + 迁移截止时间 |
