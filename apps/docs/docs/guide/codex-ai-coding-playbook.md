# Codex 与 AI 编码经验手册

## TL;DR

- 本文把你在 `one-base-template` 中已经稳定复用的 Codex/AI 编码行为，沉淀成可执行手册。
- 目标不是“写心得”，而是形成可复制的工作流：**先定边界，再并行实现，最后用证据链收口**。
- 页面会按仓库真实记录持续维护，默认参考 `.codex/operations-log.md`、`.codex/testing.md`、`.codex/verification.md` 与 `docs/plans/*.md`。

## 适用范围

- 仓库：`/Users/haoqiuzhi/code/one-base-template`
- 场景：使用 Codex 或其他 AI 编码助手进行需求拆解、实现、验证、文档同步与治理沉淀。
- 对象：个人开发者、模块 owner、负责规则治理的维护者。

## 背景与目标

你在实际使用里已经形成一套高一致性的工程习惯：写计划、收口范围、分阶段验证、失败可追溯、证据可落盘。  
本手册把这些“已发生的高频行为”结构化，便于两类复用：

1. 你自己后续开工时，快速复用同一套高质量动作。
2. 团队协作时，减少“口头经验”，转成统一执行口径。

## 已沉淀的 10 条实践

| 编号 | 实践名称           | 你已经稳定执行的动作                                                             |
| ---- | ------------------ | -------------------------------------------------------------------------------- |
| 1    | 范围冻结优先       | 开始前明确 `目标/边界/非范围/DoD`，避免任务中途膨胀。                            |
| 2    | Skill 链路先行     | 先定义技能执行顺序，再进入实现，降低“凭记忆拍脑袋”。                             |

## 可直接复用的执行模板

### 模板 A：需求启动模板（10 分钟）

1. 冻结范围：写清目标、非范围、验收口径。
2. 选执行链：决定先用哪些 skill，再开始实现。
3. 产出计划：按 `Chunk/Task` 拆分，单步可在 2-5 分钟完成。
4. 设验证最小集：按影响面列出 `typecheck/lint/build/test`。

### 模板 B：实现与验证模板（循环）

1. 先让用例或门禁失败（RED）。
2. 做最小修复并复跑（GREEN）。
3. 把失败原因、修复点、验证命令写入记录。
4. 每个里程碑结束同步 docs，保证规则与实现一致。

### 模板 C：收口模板（发布前）

1. 复核改动是否越界（是否触碰非范围）。
2. 复核证据是否完整（命令、结果、阻断项归因）。
3. 复核文档入口是否可达（导航、总览、专题页一致）。
4. 复核是否留下可维护资产（计划、规范、验证结论）。

## 维护与更新机制

### 更新触发条件

1. 你在工作方式上形成新的稳定动作（至少重复 3 次）。
2. 新增了可复用的验证门禁或证据链脚本。
3. 提问偏好出现明显变化（例如从实现转向治理、从单模块转向跨模块）。

### 更新责任

- 主维护：当前仓库维护者
- 协作维护：发起需求的模块 owner 与执行人

### 最小更新步骤

```bash
pnpm -C apps/docs lint
pnpm -C apps/docs build
```

通过后同步更新：

- `apps/docs/docs/guide/index.md`（总览入口）
- `apps/docs/docs/guide/governance.md`（治理专题地图）
- 必要时补充 `.codex/operations-log.md`、`.codex/testing.md`、`.codex/verification.md`
