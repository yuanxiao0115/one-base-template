# P4 路线（独立开发）

## TL;DR

- P4 的核心是“稳定交付”，不是“能跑就行”。
- 每次改动都要同时满足：模块契约正确、规则门禁通过、文档同步。
- 开发顺序建议固定：`模块系统 -> 菜单路由 -> 布局实践 -> 验证命令`。

## 适用范围

- 你已能独立完成基础页面开发。
- 你希望减少返工，提升一次性交付成功率。
- 你开始承担模块级改造或重构任务。

## 前置条件

1. 已完成 P2 的完整闭环（至少做过一次模块新增并通过验证）。
2. 熟悉 `apps/admin-lite` 或 `apps/admin` 的模块目录结构。
3. 能执行 `lint:arch` 与测试命令，并理解失败信息定位方式。

## 1 个迭代目标

1. 独立完成一个模块新增或改造需求。
2. 不突破架构红线（模块边界、路由 meta、组件基线）。
3. 代码与文档同步更新，避免“实现变更但文档滞后”。

## 最短执行路径

### 1. 必读主线

1. [模块系统（最终版）](/guide/module-system)
2. [菜单与路由规范（Schema）](/guide/menu-route-spec)
3. [布局与菜单](/guide/layout-menu)
4. [admin-lite Agent 红线](/guide/admin-lite-agent-redlines)
5. [开发规范与维护](/guide/development)

### 2. 按任务类型进入

1. 新增模块：先看 [模块系统（最终版）](/guide/module-system)，再看 [admin-lite 后台基座](/guide/admin-lite-base-app)。
2. 改路由/权限：先看 [菜单与路由规范（Schema）](/guide/menu-route-spec)，重点校验 `meta.access` 与 route-policy。
3. 改列表交互：先看 [布局与菜单](/guide/layout-menu)，保持 `ObTable` 编排一致。
4. 提测与合并：按 [开发规范与维护](/guide/development) 执行验证清单。

### 3. 改造完成后跑默认门禁

```bash
pnpm -C apps/admin-lite typecheck
pnpm -C apps/admin-lite lint
pnpm -C apps/admin-lite lint:arch
pnpm -C apps/admin-lite test:run
pnpm -C apps/admin-lite build
pnpm -C apps/docs lint
pnpm -C apps/docs build
```

如果你改的是 `apps/admin`，将命令中的 `admin-lite` 切换为 `admin`。

## 验证与验收

通过标准：

1. 模块功能可运行，且未破坏路由与权限行为。
2. `typecheck/lint/lint:arch/test/build` 全部通过。
3. 涉及规则或流程变更时，已同步更新 docs 页面。

## FAQ

### 为什么 P4 要强制跑 `lint:arch`？

因为 P4 阶段最常见返工点是边界越界，`lint:arch` 能在提交前发现模块耦合问题。

### 我只改了样式，也要跑全套命令吗？

至少跑 `typecheck + lint + build`。如果涉及模块或路由行为，补跑 `lint:arch` 和定向测试。

### 文档需要更新到什么程度？

只要改动影响“路径、命令、规则、行为”四类信息之一，就应同步更新对应 docs 页面。

## 升级到 P6 的信号

满足以下 2 项后，建议进入 [P6 路线（架构治理）](/guide/levels/p6)：

1. 你能判断“规则应该沉淀在根 AGENTS 还是子项目 AGENTS”。
2. 你能把一次问题修复沉淀为团队可复用的规则与验证流程。

如果当前任务只是单模块实现，优先继续留在 P4，不要过早切到治理路径。

## 回看 P2 的信号

出现以下情况时，建议回看 [P2 路线（上手）](/guide/levels/p2)：

1. 连续两次卡在基础启动或命令执行阶段。
2. 无法说清模块入口、路由规范与验证命令分别在哪一页。
