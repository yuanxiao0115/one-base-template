# P6 路线（架构治理）

## TL;DR

- P6 关注的是“团队长期可维护性”，不是单次需求交付。
- 规则必须有主版本、作用域、验证门禁和追溯证据，缺一不可。
- 推广前先做全链路校验，确保不会把治理动作变成新的不稳定源。

## 适用范围

- 你负责技术基线、评审标准、文档治理或发布策略。
- 你需要回答“为什么这样设计、如何验证、出现问题怎么回退”。
- 你正在推动跨模块规则统一或流程收口。

## 前置条件

1. 已具备 P4 交付经验，能独立完成模块级改造。
2. 熟悉根 `AGENTS.md` 与目标子项目 `AGENTS.md` 的作用域边界。
3. 能执行并解读全链路验证命令（命名、架构、测试、构建、文档）。

## 治理目标

1. 规则有主版本，且作用域清晰（根 / app / package）。
2. 关键能力有门禁（`lint:arch`、测试、构建、docs 构建）。
3. 团队可复述决策原因，不依赖口头记忆。

### 2. 治理动作前先做 5 项检查

1. 是否存在重复规则、冲突规则、无主版本规则。
2. 规则是否落在正确作用域（根 / app / package）。
3. 代码、测试、文档是否同步更新。
4. 新增能力是否有可回归命令与通过标准。
5. 是否降低了 P2/P4 的心智负担。

### 3. 推广前执行基线命令

```bash
pnpm check:naming
pnpm lint:arch
pnpm test:run
pnpm typecheck
pnpm lint
pnpm build
pnpm check:admin:bundle
pnpm check:admin-lite:bundle
pnpm -C apps/docs lint
pnpm -C apps/docs build
```

## 验证与验收

通过标准：

1. 规则文本、代码实现、测试门禁、文档入口四者一致。
2. 全链路命令通过，或失败项有明确“非阻断 + 后续计划”记录。
3. 本轮治理有可追溯证据（计划、操作日志、验证记录、提交记录）。

## FAQ

### 为什么 P6 强调“主版本规则”而不是就地补注释？

因为没有主版本会导致同类规则在不同文档中漂移，后续执行口径不一致。

### 治理改造必须一次全量完成吗？

不需要。优先做“关键路径最小闭环”，再分批推进，避免大改带来的回归风险。

### 如果全链路命令里有历史债务失败怎么办？

要在验证记录里明确“失败原因、影响范围、与本次改造关系、后续计划”，不能直接忽略。

## 产出要求

1. 每次治理改动都附“影响范围 + 风险 + 验证证据”。
2. 优先给 P2/P4 提供可执行路径，不把治理复杂度下压给所有开发者。
3. 重要架构决策必须在 docs 可检索、可复述、可回归。

## 回看 P4/P2 的信号

出现以下情况时，建议暂时回看低一层路径：

1. 本次任务仅涉及单模块实现、页面交互或局部路由修复：回看 [P4 路线（独立开发）](/guide/levels/p4)。
2. 团队成员仍无法稳定完成首轮启动与基础验证：回看 [P2 路线（上手）](/guide/levels/p2) 先补基础闭环。
3. 治理方案难以解释给非维护角色：先把规则降维到 P4 可执行语言，再继续治理推进。
