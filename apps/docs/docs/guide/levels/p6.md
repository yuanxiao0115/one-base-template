# P6 路线（架构治理）

## 适合对象

- 负责技术基线、评审标准、文档治理或发布策略。
- 关注长期可维护性、团队协作效率与变更风险。

## 治理目标

1. 规则有主版本、作用域清晰、可执行可验证。
2. 关键能力有门禁（lint:arch / test / bundle / docs build）。
3. 团队能按统一心智回答“为什么这样设计”。

## 架构治理检查清单

1. 是否存在重复规则、冲突规则、无主版本规则。
2. 规则是否落在正确作用域（根 / app / package）。
3. 代码、测试、文档是否同步（避免“代码变了文档没变”）。
4. 新增能力是否有可回归命令与通过标准。
5. 是否降低了 P2/P4 的心智负担，而不是增加理解门槛。

## 推广前基线命令

```bash
pnpm check:naming
pnpm lint:arch
pnpm test:run
pnpm typecheck
pnpm lint
pnpm build
pnpm check:admin:bundle
pnpm check:admin-lite:bundle
pnpm -C apps/docs lint
pnpm -C apps/docs build
```

## 产出要求

1. 每次治理改动都附“影响范围 + 风险 + 验证证据”。
2. 为 P2/P4 提供可执行路径，不把深度规则直接下压给所有人。
3. 重要架构决策在 docs 可追溯，且能被团队复述。
