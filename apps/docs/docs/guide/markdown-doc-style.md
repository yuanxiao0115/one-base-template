# Markdown 技术文档规范

<div class="doc-tldr">
  <strong>TL;DR：</strong>先写结构，再写内容，最后做排版与验收。技术文档至少包含目标、范围、步骤、验证与 FAQ，确保“可读、可执行、可维护”。
</div>

## 背景与目标

当前仓库文档已覆盖架构与工程实践，但不同页面在结构与排版上仍有差异。
本规范用于统一 Markdown 技术文档的写法，降低协作沟通成本，减少“看懂了但做不出来”的情况。

目标：

- 统一页面信息架构，保证读者快速定位关键信息。
- 统一排版细节，提升阅读体验与站点视觉一致性。
- 统一验收口径，让文档交付可验证、可回归。

## 适用范围与非范围

### In Scope

- `apps/docs/docs/**` 下的指南页、规范页、操作手册。
- 仓库级 `README.md`、实施方案、迁移说明、发布 SOP。
- 需要命令、配置、代码示例的技术文档页面。

### Out of Scope

- 营销文案、公告文案、活动介绍类页面。
- 与仓库无关的外部产品说明文档。
- 自动生成的 API 参考文档（若由工具全量生成）。

## 前置条件

- 明确目标读者：新同学 / 日常开发 / 维护负责人。
- 明确页面目的：介绍概念、指导操作或沉淀规范。
- 明确可信信息源：代码路径、命令、配置文件、既有规范。

## 标准写作流程

### 1. 定义页面定位

先用 1-2 句写清“这页解决什么问题”，再列出适用范围。

### 2. 搭建固定骨架

至少包含以下章节：

1. 背景与目标
2. 适用范围（In Scope / Out of Scope）
3. 前置条件
4. 操作步骤
5. 验证与验收
6. FAQ

### 3. 填写可执行内容

每个操作步骤都补齐：

- 操作动作
- 执行命令
- 预期结果
- 失败处理（如有）

### 4. 统一视觉与术语

- 标题层级控制在 `#` 到 `###`。
- 代码块必须声明语言。
- 参数、错误码、状态优先使用表格。
- 同一页面术语保持一致，不混用同义词。

### 5. 验收并发布

文档改动完成后，在仓库根目录执行：

```bash
pnpm -C apps/docs lint
pnpm -C apps/docs build
```

通过标准：命令退出码为 `0`，且页面链接、目录、样式显示正常。

## 推荐页面骨架模板

```md
# 页面标题

> TL;DR：一句话说明价值与验收方式。

## 背景与目标

## 适用范围

## 前置条件

## 操作步骤

## 验证与验收

## FAQ
```

## 排版规范速查

| 项目     | 规范             | 说明                      |
| -------- | ---------------- | ------------------------- |
| 标题层级 | 不超过三级       | 避免目录过深影响阅读      |
| 段落长度 | 3-5 行           | 一段一个核心观点          |
| 代码块   | 必须标注语言     | `bash` / `ts` / `json` 等 |
| 强调方式 | 关键字加粗       | 避免整段加粗              |
| 表格场景 | 参数/状态/错误码 | 对比信息优先表格          |

## 文件引用规则（避免本机路径泄漏）

- 禁止在文档中写本机绝对路径（例如 `~/workspace/...`、`C:\\workspace\\...`）。
- 文件路径统一写仓库相对路径（如 `apps/admin/src/config/app.ts`）。
- 文件引用默认保留仓库相对路径全文，便于直接阅读与全文检索。
- 命令示例统一使用相对路径，例如：`pnpm -C apps/admin lint`、`pnpm -C apps/docs build`。

推荐写法：

```md
请修改 `apps/admin/src/bootstrap/index.ts` 与 `packages/app-starter/src/startup.ts`。
```

## 常见问题（FAQ）

| 问题                               | 原因                         | 处理方式                         |
| ---------------------------------- | ---------------------------- | -------------------------------- |
| 页面内容很多但读者反馈“找不到重点” | 章节顺序按作者思路而非任务流 | 按“问题 -> 步骤 -> 验收”重排章节 |
| 文档看起来杂乱                     | 样式与术语不统一             | 统一标题、列表、表格与术语规范   |
| 命令执行失败                       | 缺少执行目录或前置条件       | 在命令前补充执行上下文与依赖     |
