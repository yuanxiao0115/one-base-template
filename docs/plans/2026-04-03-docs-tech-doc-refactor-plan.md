# docs 技术文档分析与改造方案（2026-04-03）

> 日期：2026-04-03  
> 作用域：`apps/docs/docs/**`、`apps/docs/docs/.vitepress/**`  
> 执行顺序：`brainstorming -> technical-doc-collaboration -> write-markdown-tech-docs -> vitepress-doc-beauty`

## 背景

本轮目标是先对 `apps/docs` 技术文档做系统分析，再进行结构化改造，提升三件事：

1. 新同学首跑成功率（能否只看文档独立完成）
2. 规则可检索性（能否快速定位“该改哪里、该跑什么”）
3. 文档一致性（页面骨架、导航入口、视觉风格是否统一）

## 受众画像（Audience First）

| 角色         | 先验知识                         | 目标任务                        | 成功标准                       | 失败成本                 |
| ------------ | -------------------------------- | ------------------------------- | ------------------------------ | ------------------------ |

## 现状审计（问题框架）

### 1) 结构化指标（`apps/docs/docs/guide/*.md`）

- 总页面数：`45`
- 含 TL;DR：`14`
- 含“适用范围”：`22`
- 含“步骤/快速路径”：`4`
- 含“验证/验收”：`27`
- 含 FAQ：`8`

结论：当前文档“概念说明”较多，但“最小可运行路径 + 失败处理 + 维护触发条件”覆盖不足。

### 2) 关键痛点

1. **入口页与规范页之间缺少“分析 -> 改造 -> 验收”的统一主线**。
2. `quick-start`、`architecture`、`development` 的页面骨架不一致，读者在不同页面切换时心智跳变明显。
3. 维护治理入口已有较多页面，但缺少“文档改造方法论”的专页，导致经验主要留在会话而非文档。

### 3) 风险与限制

- 不进行全量 45 页重写，避免一次性大改带来信息回归风险。
- 本轮优先改“入口链路关键页 + 方法论页 + 导航联动”，采用最小差异改造策略。

## 方案对比（Brainstorming）

### 方案 A：全量页面统一模板（45 页一次改完）

优点：结构最统一。  
缺点：改动过大，回归成本高，容易引入过期信息。

### 方案 B：只改视觉，不改内容结构

优点：投入小。  
缺点：无法提升“可执行性”，问题本质不变。

### 方案 C：关键链路优先 + 模板化方法沉淀（本轮采用）

优点：

1. 先提升高频入口页质量（收益立竿见影）。
2. 新增方法论页面，为后续批量改造提供统一模板。
3. 同步导航与视觉细节，保证“找得到 + 看得懂 + 看着顺”。

缺点：

- 其余长尾页面仍需后续批次收口。

## 本轮改造范围（In Scope）

1. 新增文档改造方法页（分析结果、执行步骤、验收清单、维护职责）。
2. 重写以下关键页，统一骨架（TL;DR、范围、前置条件、步骤、验证、FAQ）：
   - `guide/quick-start.md`
   - `guide/architecture.md`
   - `guide/development.md`
3. 同步 `guide/index.md` 与 `.vitepress/config.ts` 导航入口。
4. 主题样式做轻量增强，提升信息卡片与首屏阅读层次。

## 非范围（Out of Scope）

1. 全量重写 45 个技术页。
2. 修改业务代码与运行时逻辑。
3. 引入新的文档构建插件或外部搜索服务。

## 最小可运行路径（本轮执行）

在仓库根目录执行：

```bash
pnpm -C apps/docs lint
pnpm -C apps/docs build
```

通过标准：

1. `lint` 无 warning/error。
2. `build` 成功且新增页面可在导航中访问。

## 验收清单

1. 关键页具备统一骨架，读者可 3 次点击内到达高频任务入口。
2. 新增“文档改造方法页”与总览互链，避免孤立页面。
3. 顶部导航入口仍不超过 7 个，侧边栏分组保持任务流组织。
4. 样式调整仅限 token 与信息层次，不破坏现有阅读体验。
